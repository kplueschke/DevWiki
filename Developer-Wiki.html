Git & Versionskontrolle

BlameZeigt für jede Codezeile an, welcher Autor sie wann zuletzt geändert hat.
BranchEin isolierter, paralleler Entwicklungszweig für Features oder Bugfixes.
Cherry-PickEinen ganz gezielten, einzelnen Commit aus einem anderen Branch übernehmen.
CommitEin gespeicherter Schnappschuss (Snapshot) von Code-Änderungen inkl. Nachricht.
ConflictTritt auf, wenn zwei Personen dieselbe Zeile unterschiedlich geändert haben (Merge-Konflikt).
Detached HEADZustand, in dem kein Branch ausgewählt ist, sondern direkt auf einem Commit gearbeitet wird.
Fast-ForwardEin Merge, bei dem der Branch-Zeiger einfach auf den neuesten Commit verschoben wird.
FetchRuft Informationen über Änderungen vom Server ab, ohne sie lokal zusammenzuführen.
ForkEine vollständige Kopie eines Repositories auf dem Server (oft für Open Source Beiträge).
Git HooksSkripte, die automatisch bei Ereignissen (z. B. vor einem Commit) ausgeführt werden.
MergeDas Zusammenführen von Änderungen aus verschiedenen Zweigen zu einem gemeinsamen Stand.
Origin / UpstreamStandardname für den Hauptserver (Remote) vs. das Original-Repo bei einem Fork.
PullKombination aus Fetch und Merge: Änderungen laden und sofort lokal einspielen.
Pull Request (PR)Förmlicher Vorschlag, Code-Änderungen zu prüfen und in einen Hauptzweig zu mergen.
PushLokale Commits auf den entfernten Server (Remote) hochladen.
RebaseDie Basis eines Branches auf einen neuen Startpunkt setzen, um eine lineare Historie zu erhalten.
RepositoryDas gesamte Projektverzeichnis inklusive der kompletten Versionshistorie.
SquashMehrere kleine Commits zu einem einzigen, sauberen Commit zusammenfassen.
StashAktuelle, unfertige Änderungen kurzzeitig in einen Zwischenspeicher ("Parkplatz") legen.
TagEin fester, benannter Marker in der Historie, meist zur Kennzeichnung von Release-Versionen.

Integration TestÜberprüft das korrekte Zusammenspiel zwischen zwei oder mehr Modulen oder Systemen.

ProfilingAnalyse der Laufzeit und des Speicherverbrauchs, um Performance-Engpässe zu finden.
Regression TestWiederholter Test nach Änderungen, um sicherzustellen, dass alte Funktionen noch arbeiten.
Smoke TestEin schneller Oberflächentest, um zu prüfen, ob die Basis-Funktionen überhaupt starten.
Snapshot TestVergleicht die aktuelle UI oder Datenstruktur mit einem gespeicherten "Soll-Zustand".
StubEin minimaler Platzhalter, der auf Aufrufe lediglich mit festen Werten antwortet.
TDDTest Driven Development: Der Test wird vor dem eigentlichen Funktionscode geschrieben.
Unit TestAutomatisierter Test der kleinstmöglichen Einheit (z. B. einer einzelnen Methode).
WatchÜberwachungstool im Debugger, das den Wert bestimmter Variablen permanent anzeigt.
